**Chapter 2: Working with Text Data**
- Embedding: Converting data into a vector format
- For the training data for the LLM, we are using "The Verdict" a short story by Edith Wharton.
    - Goal is to tokenize this entire text file into individual words and special characters and then turn them into embeddings for LLM training
- First, we are simply looking at sample texts and figuring out how to properly split the text to include what we want
- After loading the short story in, we split all the text by characters and spaces
- Now that the tokens are created, we need to create token IDs. For that, something simple could work like assinging each of them numbers
- For encoding that would work. For example, in the sentence, "Hello World. This is Rino.", "Hello" would be 0, "World" would be 1 and so on. We store this in a dictionary.
- For decoding, we create an inverse of the dictionary and this will allow us to decode the token IDs and form a proper sentence with that.
- The way this works is it basically fetches the word from the dictionary that we have and join them into a proper sentence. 

- New Tokenizer:
    - In the old tokenizer, the tokenizer was unable to use unknown words that it wasn't already trained on. For example, the word Hello was not present in the short story. So when trying to tokenize the sentence with the unknown word in that, it would throw an error because the tokenizer was unable to find the word, integer pair in the dictionary. So in the new tokenizer, so help with this, we added two new tokens to identify the end of the text and for unknown words. The two new tokens being <|unk|> and <|endoftext|>. But the problem that comes with this is that we don't know which word each <|unk|> represents. All we know is that it wasn't in the training data. To solve this proble, we can use Byte Pair Encoding, which breaks up unknown words into smaller words.

- Byte Pair Encoding: The byte pair encoding algorithm works by approaching unknown workds by splitting them into individual characters and subwords and tokenize them. So when the tokenizer approaches an unknown word, it can tokenize it by splitting it up into smaller subword units and individual characters. The algorithm creates its vocabulary by adding individual characters first, and then combining the letters that appear most frequently. Then those subwords are merged into words. This process happens iteratively.

- The next step to create the embeddings is to generate the input-target pairs. The need for the input-target pairs is because LLMs learn to predict words one at a time. So by creating input target pairs, we can train the LLM to predict one word at a time. The way we do this is a method called "sliding window". Basically what happens is that you create the training data into chunks, with the target output being the next token outside of the input chunk. For example, in the textbook, the input chunk could be "LLMs learn to predict one word at a" and the output would "learn to predict one word at a time". By comparing these two chunks, the LLM learns to predict the next word in the sentence. To efficiently achieve this approach, we use a dataloader and the dataset class from pytorch. Using these, we can create tensors of rows of input contexts from the provided text. We also create tensors of rows of output targets from the contexts, just shifted by an arbitrary amount. This shift is known as stride. The input parameters for the dataloader class we created were:
    - Text: The text we want tokenized
    - Batch_size: The number of samples for the inputs and outputs
    - max_length: The number of tokens in each of the tensors
    - stride: By how much the "sliding window" moves from the input to the output. Ex: If the input tensor is tokens 1, 2, 3, 4; the ouput tensor would be tokens 3, 4, 5, 6
    - shuffle: If you want to shuffle the order of the rows in the input
    - num_workers: The number of CPU process used for the preprocessing tasks
The best practice for the max_length and the stride is to set them equal to each other because this way, you reduce chances for overfitting and allow for each token to be utilized in the training data and not allow for any token to be skipped. 

- The next step in this process would be to actually create the embeddings for the tokens. The LLM cannot train itself on the raw tokens or the token ids that represent those raw tokens. Instead, the LLM uses a continuous vector representation, or embedding, so that the LLM can process the vectors and use is accordingly for back propagation.
- The embedding layer, which we use pytorch for, is basically a table that allows us to look up a token's embedding weight. The weight matrix is initially populated with random weights, which is then tweaked throughout the LLM's learning process. The input parameters for the embedding class we created were the vocab_size and the output_dim. The vocab_size represents the total number of unique tokens in the tokenizer's vocabulary; This is the number of rows in the weight matrix. The output_dim represents the number of columns in the embedding weight matrix. 
    - Another problem that we need to encounter is that the token embeddings and the word positions themselves dont have any way of determining the embedding of the token in regards to its position in the input sentence. This is because the LLM's are position agnostic, specifically the self-attention mechanism. So for example, if the word fox appeared in the first or last word of a sentence, the standard embedding layer will map it as the same vector even though technically its not the same vector. 
    - To solve this, there are two embeddings: relative positional embeddings and absolute positional embeddings. The absolute positional embeddings are directly associated with the specific positions in a sequence. For each token's embedding, it will also have a unique embedding that represents the token's position within a sequence. Relative positional embeddings are where the LLM doesn't focus on the absolute position of a token but rather, it focuses on the distance between the tokens, or the relative position. This allows for better generalization to various lengths. 
    - **The final Input embedding** is a combination of the positional embedding and the standard embedding layer. The positional embedding instead of looping up the word ID or the token ID, it looks up based on its position in the sentence and the standard embedding layer looks up using the word ID or the token ID. The combination of both of these are what result in the final input embedding.
